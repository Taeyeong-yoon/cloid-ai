섹션 A

개념 분리 인터랙티브 맵

6가지 핵심 개념을 카드로 눌러 비교하세요. 이것들은 모두 다른 레이어에 있습니다.

🧩 Skill은 프롬프트, Plugin은 패키지, Marketplace는 배포 채널이다
❌ 틀리기 쉬운 표현 보기 (8가지) ▼
❌ "플러그인 = 스킬이다"
⭕ 스킬은 마크다운 지시문이고, 플러그인은 스킬 + 커넥터 + 명령어를 묶은 패키지입니다. 스킬은 플러그인의 재료 중 하나입니다.
❌ "마켓플레이스는 앱스토어처럼 돈을 내고 사는 곳이다"
⭕ 현재 Claude 마켓플레이스는 유료 판매 기능이 없습니다. 플러그인 배포/관리 채널에 가깝습니다.
❌ "Cowork과 claude.ai는 같은 것이다"
⭕ claude.ai는 채팅 인터페이스이고, Cowork은 비개발자가 복잡한 업무를 AI에게 맡기는 에이전트 작업 환경입니다.
❌ "커넥터 = MCP 서버이다"
⭕ 커넥터는 MCP 서버를 사용자 친화적으로 포장한 것입니다. 내부적으로 MCP를 쓰지만 사용자는 설정 파일을 직접 만질 필요 없이 토글로 켜고 끕니다.
❌ "플러그인은 개발자만 만들 수 있다"
⭕ 플러그인은 마크다운 파일 폴더입니다. Claude에게 "영업 플러그인 만들어줘"라고 말하면 대화로 만들 수 있습니다.
❌ "프라이빗 마켓플레이스는 비용이 추가로 든다"
⭕ Team/Enterprise 플랜에 포함된 기능입니다. 플러그인 자체에 추가 비용은 없습니다(외부 서비스 연결 시 해당 서비스 비용은 별도).
❌ "Claude Code 플러그인과 Cowork 플러그인은 완전히 다르다"
⭕ 같은 플러그인 구조(폴더 + 마크다운)를 공유합니다. 다만 실행 환경(CLI vs 웹)에 따라 체감이 다를 수 있습니다.
❌ "스킬은 대단한 기술이다"
⭕ 본질적으로 재사용 가능하게 저장해둔 고급 프롬프트입니다. 거기에 코드 실행 권한이나 MCP 연결을 붙일 수 있을 뿐입니다.
섹션 B

전체 구조: 레이어별 이해

아래에서 위로 쌓이는 구조를 한 화면에서 읽을 수 있게 정리했습니다. 각 레이어는 아래층을 재료로 삼아 위층에서 더 큰 실행 단위가 됩니다.

아래층은 재료, 위층은 실행과 배포

맨 아래의 Skill은 지시문이고, Connector는 데이터 연결입니다. 이 둘을 묶어 Plugin이 되고, 그 Plugin을 팀에 퍼뜨리는 곳이 Marketplace이며, 실제로 쓰는 자리가 Cowork / Claude Code입니다.

Skill → Connector → Plugin → Marketplace → Runtime
📝 Skill
가장 작은 단위
Claude에게 "이 상황에서는 이렇게 일하라"라고 알려주는 지시문입니다. 반복되는 업무 방법을 문서로 저장한 단계입니다.
무엇을 담나

업무 목적, 진행 순서, 출력 형식, 금지 사항, 예시 답변 같은 작업 규칙을 담습니다.

예시

`회의록 요약 스킬`, `영업 미팅 준비 스킬`, `블로그 초안 작성 스킬`처럼 반복 업무 하나를 정의합니다.

↑ 이 스킬이 실제 데이터를 쓰려면 연결이 필요
🔌 Connector
외부 데이터 연결
Claude가 Gmail, Slack, Google Drive, GitHub 같은 외부 서비스에 접근하게 해 주는 연결부입니다. 지시문만으로는 못 보는 데이터를 여는 단계입니다.
왜 필요한가

실제 메일, 파일, 일정, CRM 데이터가 있어야 업무 자동화가 현실적인 결과를 만들 수 있습니다.

예시

`Gmail 연결`, `Google Calendar 연결`, `Salesforce 연결`을 켜 두면 Claude가 실제 업무 데이터를 읽습니다.

↑ 지시문과 연결을 실전에 맞게 한 상자로 묶음
📦 Plugin
실행 패키지
Skill과 Connector, 필요하면 슬래시 명령과 설정까지 함께 넣은 패키지입니다. "영업용 Claude", "재무용 Claude"처럼 역할별 전문가 세트를 만드는 단계입니다.
안에 들어가는 것

스킬 여러 개, 커넥터 설정, `/call-prep` 같은 명령어, 역할 설명, 폴더 구조와 설정 파일이 함께 들어갑니다.

예시

`Sales Plugin` 안에 고객 조사 스킬, CRM 커넥터, 미팅 준비 명령을 함께 묶어 둡니다.

↑ 좋은 패키지를 여러 사람이 쓰려면 배포 채널이 필요
🏪 Marketplace
배포와 관리
플러그인을 개인 파일로 끝내지 않고 팀과 조직에 배포하는 채널입니다. 어떤 플러그인을 누가 설치하고 어떤 버전을 쓰는지 관리하는 운영 레이어입니다.
하는 일

플러그인 공개 범위 관리, 버전 업데이트, 조직 내 자동 설치, GitHub 기반 배포를 담당합니다.

예시

교육팀이 만든 `신입 온보딩 플러그인`을 사내 마켓플레이스에 올려 전 직원이 같은 버전으로 쓰게 합니다.

↑ 배포된 패키지는 실제 작업 환경에서 실행됨
🖥️ Cowork / Claude Code
실행 환경
같은 플러그인도 어디서 실행하느냐에 따라 체감이 달라집니다. Cowork은 업무 위임 중심, Claude Code는 로컬 개발 실행 중심입니다.
Cowork

비개발자가 버튼, 폼, 워크플로 중심으로 업무를 맡기고 결과를 받는 환경입니다.

Claude Code

개발자가 터미널, 파일 수정, 셸 명령, git 흐름까지 직접 다루는 실행 환경입니다.

한 문장 요약

Skill은 행동 규칙, Connector는 데이터 연결, Plugin은 실행 패키지입니다.

팀 관점

Marketplace는 좋은 Plugin을 팀 전체에 복제 가능하게 만드는 배포 시스템입니다.

사용자 관점

최종 사용자는 Cowork 또는 Claude Code에서 이 구조를 실제 업무나 개발 흐름으로 체감합니다.

💡 핵심: Skill 하나만으로도 충분히 유용합니다. Plugin은 여러 Skill을 묶어 더 강력하게 만든 것이고, Marketplace는 그걸 팀에 배포하는 수단입니다. 필요에 따라 단계적으로 확장하세요.
섹션 C

플러그인 없을 때 vs 있을 때

"매번 설명하는 AI"와 "이미 전문가인 AI"의 차이를 체감하세요.

플러그인이 실제로 일하는 상세 흐름도

플러그인이 없으면 사용자가 매번 역할, 규칙, 데이터 소스를 반복 설명해야 합니다. 플러그인이 있으면 그 맥락과 명령, 연결부가 한 패키지 안에 들어가서 바로 실행 가능한 업무 흐름으로 바뀝니다.

반복 설명 → 패키지 실행
플러그인 없을 때톤, 형식, 회사 절차, 데이터 소스를 요청할 때마다 다시 알려줘야 해서 결과가 흔들립니다.
플러그인 내부Skill이 역할과 규칙을 잡고, Commands가 호출 방식을 고정하고, Connector가 외부 도구 연결을 담당합니다.
플러그인 있을 때사용자는 짧은 목표만 말하고, 실행 환경은 Cowork든 Claude Code든 같은 패키지를 불러 일관된 결과를 만듭니다.

섹션 G

첫 실습 실험실

스킬 만들기 → 플러그인 조립 → 설치까지 단계별로 따라하세요.

🎯 목표: 나만의 스킬을 만들고, 플러그인으로 묶어서 실제로 사용해본다

✅ 실습 체크리스트

  • 스킬의 목적(어떤 업무를 반복하는지)을 정했는가
  • SKILL.md 파일의 구조(프론트매터 + 본문)를 이해했는가
  • 스킬을 실제로 만들어봤는가 (직접 작성 또는 Claude에게 요청)
  • 플러그인 폴더 구조를 이해했는가
  • 플러그인을 설치하고 실제로 작동하는 것을 확인했는가
⚠️ 자주 발생하는 실패 원인 ▼
  • description 누락 — 프론트매터에 description이 없으면 Claude가 언제 쓸지 판단하지 못합니다
  • 폴더 구조 오류 — SKILL.md가 skills/ 하위의 폴더 안에 있어야 합니다
  • 코드 실행 미활성화 — Settings → Capabilities에서 Code execution이 꺼져 있으면 스킬이 작동하지 않습니다
  • ZIP 구조 문제 — 웹에서 업로드 시 폴더가 ZIP의 루트여야 합니다 (하위 폴더 안에 중첩되면 안 됨)
  • 커넥터 인증 실패 — 외부 서비스 커넥터 사용 시 인증이 만료되었을 수 있습니다

섹션 K

복사용 메모 / 실행 체크리스트

바로 실행에 옮길 수 있도록 복사해서 사용하세요.

📋 내가 지금 이해한 핵심

• Skill = 재사용 가능한 고급 프롬프트 (SKILL.md 파일)
• Plugin = Skill + Connector + 슬래시 명령을 묶은 패키지
• Connector = 외부 도구 연결 (MCP 기반, 토글로 켜고 끔)
• Marketplace = 플러그인을 팀에 배포하는 채널
• Cowork = 비개발자용 AI 에이전트 작업 환경
• Plugin은 마크다운 파일 폴더일 뿐이다
• 개인은 Skill만으로도 충분하다
• 팀 배포가 필요하면 Plugin → Marketplace 순으로 확장

📋 첫 스킬 만들기 체크리스트

□ 반복하는 업무/프롬프트 1개 선정
□ SKILL.md 파일 작성 (또는 Claude에게 요청)
  - name: 스킬 이름
  - description: 언제 사용할지 설명
  - 본문: Claude가 따를 지시사항
□ Settings → Capabilities → Code execution 활성화 확인
□ Settings → Customize → Skills에서 업로드
□ 관련 요청을 해보고 자동으로 스킬이 적용되는지 확인
□ 결과가 기대와 다르면 description과 본문 수정

📋 플러그인으로 확장하기 체크리스트

□ 묶을 Skill 2개 이상 준비
□ 필요한 Connector (외부 서비스) 확인
□ 플러그인 폴더 구조 생성:
  my-plugin/
  ├── .claude-plugin/plugin.json  (이름, 설명, 버전)
  ├── .mcp.json                   (커넥터 설정)
  ├── skills/                     (스킬 폴더들)
  └── commands/                   (슬래시 명령)
□ 슬래시 명령 테스트 (/명령이름)
□ 팀 배포 필요 시: GitHub 레포 → Marketplace 연결
